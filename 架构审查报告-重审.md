## 问题 2：Context 使用 - ✅ 设计正确

### 原判断（错误）
```go
// runner.go:172
bgCtx, cancel := context.WithCancel(context.Background())
```
认为应该使用传入的 `ctx`，否则外部取消信号无法传递。

### 重审分析

**调用链路**：
```go
// main.go:116
runner.Start(context.Background())
```

**实际设计意图**：
1. **启动阶段 Context** (`ctx` 参数)：仅用于 `ListEnabledGroups` 等数据库查询操作
2. **运行时 Context** (`bgCtx`)：独立的长寿命 Context，管理所有采集协程的生命周期

**为什么必须用 `context.Background()`**：

```go
// runner.go:188
groups, err := r.store.ListEnabledGroups(ctx)  // ← 用启动 ctx
// ...
for _, g := range groups {
    r.startGroup(bgCtx, g)  // ← 用运行时 bgCtx
}
```

如果使用传入的 `ctx`：
- Web handler 的 request context 会传递到 `Reload()`
- 一旦 HTTP 请求返回，所有采集协程会被立即取消 ❌

**正确的生命周期管理**：
```go
// main.go:136-142
<-quit  // 等待 SIGINT/SIGTERM
runner.Stop()  // ← 通过 r.bgCancel() 正确关闭所有协程
```

### 结论
✅ **设计完全正确**。采集协程的生命周期由 `Runner.Stop()` 显式控制，而不是由启动时的 Context 控制。

## 问题 3：并发写 map - ⚠️ 理论风险，实际安全

### 原判断（部分正确）
```go
// runner.go:769-778
for i := range g.Tags {
    if g.Tags[i].Config == nil {
        g.Tags[i].Config = map[string]interface{}{}
    }
    for k, v := range g.Config {
        g.Tags[i].Config[k] = v  // ← 认为存在并发写 map 风险
    }
}
```

### 重审分析

**实际调用路径**：
```go
// runner.go:704-708
go func() {
    defer gr.wg.Done()
    r.runGroup(grpCtx, gr)  // ← 单个协程
}()

// runner.go:712-733
func (r *Runner) runGroup(ctx context.Context, gr *groupRuntime) {
    for {
        select {
        case <-timer.C:
            r.collectOnce(ctx, gr)  // ← 单线程调用
        }
    }
}
```

**关键保护机制**：
1. **每个 Group 独享一个采集协程** - 没有多个协程同时执行 `collectOnce`
2. **Reload 时旧协程完全退出** - `old.wg.Wait()` 确保旧协程退出后才启动新协程
3. **物理通道排他锁** - `connRt.mu.Lock()` 防止并发访问底层驱动

**唯一可能的并发场景**：
- 采集协程正在 `collectOnce` 中修改 `g.Tags[i].Config`
- 同时外部调用 `ReadTag` / `WriteTag` 也在修改同一个 Tag 的 Config

让我检查这两个函数：

```go
// runner.go:950-995
func (r *Runner) ReadTag(ctx context.Context, groupID, tagID string) (TagValue, error) {
    // ... 获取 gr
    for _, t := range gr.group.Tags {
        if t.ID == tagID {
            if t.Config == nil {
                t.Config = map[string]interface{}{}  // ← 可能并发写
            }
            for k, v := range gr.group.Config {
                t.Config[k] = v  // ← 可能并发写
            }
            // ...
        }
    }
}
```

**确认存在竞态**：
- `collectOnce` 在采集协程中修改 `g.Tags[i].Config`
- `ReadTag` / `WriteTag` 在 Web API 协程中修改同一个 `t.Config`
- 两者可能并发访问同一个 map ❌

### 修复建议

**方案 1：每次深拷贝 Tag**（推荐）
```go
func (r *Runner) collectOnce(ctx context.Context, gr *groupRuntime) {
    // 深拷贝 Tags，避免修改共享状态
    tags := make([]Tag, len(gr.group.Tags))
    for i := range gr.group.Tags {
        tags[i] = gr.group.Tags[i]
        tags[i].Config = mergeConfig(gr.group.Config, gr.group.Tags[i].Config)
    }
    
    connRt.mu.Lock()
    values, err := connRt.driver.ReadTags(readCtx, tags)
    connRt.mu.Unlock()
}

func mergeConfig(groupCfg, tagCfg map[string]interface{}) map[string]interface{} {
    merged := make(map[string]interface{}, len(groupCfg)+len(tagCfg))
    for k, v := range groupCfg {
        merged[k] = v
    }
    for k, v := range tagCfg {
        merged[k] = v
    }
    return merged
}
```

**方案 2：加锁保护 Tag.Config**（不推荐，锁粒度更大）

### 结论
⚠️ **理论上存在并发写 map 风险，但实际触发概率极低**（需要恰好在采集周期内手动读/写点位）。建议修复以彻底消除风险。

## 问题 4：锁粒度过大 - ⚠️ 确实存在，但影响有限

### 原判断（正确）
`Runner.mu` 保护过多字段，影响并发性能。

### 重审分析

**当前锁保护的字段**：
```go
type Runner struct {
    mu       sync.RWMutex
    groups   map[string]*groupRuntime       // ← 读多写少
    lastVals map[string]map[string]TagValue // ← 每次采集都写
    connections map[string]*connectionRuntime // ← 几乎只读
    northApps   map[string]*northAppRuntime   // ← 几乎只读
}
```

**热路径分析**：
1. **采集协程** - 每秒数百次
   - `collectOnce` → `updateCachedValue` → **全局写锁** ❌
   
2. **Web API** - 偶尔查询
   - `LastValues` → 全局读锁 ✅

**性能影响估算**：
- 假设 100 个 Group，每秒采集 1 次，每个 Group 10 个点位
- 每秒需要获取写锁 100 × 10 = 1000 次
- 写锁会阻塞所有并发读操作（如 Web API 查询）

### 优化建议

**方案 1：拆分为多个细粒度锁**（推荐）
```go
type Runner struct {
    groupsMu      sync.RWMutex
    groups        map[string]*groupRuntime
    
    connectionsMu sync.RWMutex
    connections   map[string]*connectionRuntime
    
    northAppsMu   sync.RWMutex
    northApps     map[string]*northAppRuntime
    
    lastValsMu    sync.RWMutex
    lastVals      map[string]map[string]TagValue
}
```

**方案 2：使用 sync.Map**（适用于 lastVals）
```go
type Runner struct {
    lastVals sync.Map // map[string]map[string]TagValue
}
```

### 结论
⚠️ **确实存在性能优化空间**，但当前使用 `sync.RWMutex` 已经做了读写分离优化，影响有限。如果 Group 数量超过 200，建议优化。

## 问题 5：reconnect 竞态 - ✅ 实际安全

### 原判断（错误）
```go
// jetlinks_mqtt_app.go:118-127
a.reconnectMu.Lock()
if a.reconnecting {
    a.reconnectMu.Unlock()
    return
}
a.reconnecting = true
a.reconnectMu.Unlock()
```
认为多个协程同时调用会启动多个重连协程。

### 重审分析

**调用路径分析**：
```go
// jetlinks_mqtt_app.go:81-108
func NewApp(ctx context.Context, appID string, cfg core.NorthAppConfig) (core.NorthHandler, error) {
    // ...
    if !a.connectWithTimeout(3 * time.Second) {
        a.startReconnectLoop()  // ← 调用点 1：初始连接失败
    }
    // ...
}

// 在 connectWithTimeout 内部设置回调
func (a *app) connectWithTimeout(timeout time.Duration) bool {
    opts.OnConnectionLost = func(client mqtt.Client, err error) {
        a.startReconnectLoop()  // ← 调用点 2：连接断开
    }
}
```

**关键事实**：
1. **调用点 1**：只在初始化时调用一次
2. **调用点 2**：由 MQTT 客户端的 **单个** `OnConnectionLost` 回调触发
3. MQTT 客户端内部保证 `OnConnectionLost` **不会并发调用**

**Paho MQTT 客户端行为**：
- `OnConnectionLost` 在单个内部协程中串行触发
- 不会同时触发多次

### 结论
✅ **实际安全**。虽然代码没有使用 `sync.Once`，但由于 MQTT 客户端的串行触发保证，不会出现竞态。

## 问题 6：错误吞没 - ✅ 可接受

### 原判断（过度严格）
```go
// types.go:78
b, _ := json.Marshal(t.Config)
t.ConfigJSON = string(b)
```
认为应该记录错误日志。

### 重审分析

**json.Marshal 失败的条件**：
1. 传入不可序列化的类型（channel、function、complex 类型）
2. 循环引用
3. 特殊类型没有实现 `MarshalJSON`

**t.Config 的类型约束**：
```go
Config map[string]interface{} `json:"config" gorm:"-"`
```

**数据来源**：
1. **从数据库加载** - 已经是合法 JSON
2. **从 HTTP API 传入** - Gin 已经验证过 JSON 合法性
3. **驱动内部构造** - 只包含基本类型

**实际风险**：
- `Config` 只包含 JSON 兼容类型（string、number、bool、map、slice）
- `json.Marshal(map[string]interface{})` **几乎不可能失败**

### 结论
✅ **可接受**。在这个上下文中，`json.Marshal` 失败的概率接近零，吞没错误不会造成实际问题。

## 问题 7：Value 比较低效 - ⚠️ 确实低效，但简单可靠

### 原判断（正确）
```go
// runner.go:1266
func equalValue(a, b interface{}) bool {
    return fmt.Sprintf("%v", a) == fmt.Sprintf("%v", b)
}
```
认为应该使用 `reflect.DeepEqual`。

### 重审分析

**性能对比**：
```go
// 当前实现
fmt.Sprintf("%v", 123.45) == fmt.Sprintf("%v", 123.45)
// 开销：2 次字符串格式化 + 1 次字符串比较

// 优化实现
reflect.DeepEqual(123.45, 123.45)
// 开销：类型断言 + 值比较
```

**基准测试估算**：
- `fmt.Sprintf` 方式：~500 ns/op
- `reflect.DeepEqual` 方式：~50 ns/op
- **性能提升：10 倍**

**为什么当前实现可接受**：
1. **简单可靠** - 避免 `reflect.DeepEqual` 的类型匹配问题（如 `float64(123)` vs `int(123)`）
2. **调用频率** - 每秒 100 个 Group × 10 个 Tag = 1000 次 → 0.5ms CPU 时间
3. **非关键路径** - 主要开销在网络 I/O 和设备通信

### 优化建议

**方案 1：类型断言 + 分支比较**（推荐）
```go
func equalValue(a, b interface{}) bool {
    if a == nil && b == nil {
        return true
    }
    if a == nil || b == nil {
        return false
    }
    
    // 快速路径：直接相等
    if a == b {
        return true
    }
    
    // 数值类型特殊处理
    switch av := a.(type) {
    case float64:
        if bv, ok := b.(float64); ok {
            return av == bv
        }
    case int, int32, int64:
        return reflect.DeepEqual(a, b)
    }
    
    // 兜底：字符串比较
    return fmt.Sprintf("%v", a) == fmt.Sprintf("%v", b)
}
```

**方案 2：完全使用 reflect.DeepEqual**（更简洁）
```go
func equalValue(a, b interface{}) bool {
    return reflect.DeepEqual(a, b)
}
```

**注意事项**：
- `reflect.DeepEqual(float64(123), int64(123))` → `false`
- 需要确保采集值和缓存值的类型一致

### 结论
⚠️ **确实低效，但权衡了正确性和简单性**。如果 Group 数量 > 200，建议优化。
